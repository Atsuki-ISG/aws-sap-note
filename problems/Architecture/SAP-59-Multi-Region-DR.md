# Problem Statement

【SAP-59】 北米で資産運用プラットフォームを展開する金融サービス企業は、新しいオンライン取引アプリを AWS 上で公開し、個人投資家へリアルタイムのポートフォリオ閲覧と注文機能を提供している。
顧客 SLA で可用性 99.9% を約束しているうえ、取引開始直後にアクセスが急増するため自動スケールが必須と判断している。
運用拠点は遅延の観点から us-east-1 の Amazon EC2 とし、事業継続計画に基づきアクティブ／パッシブ形式の待機系を us-west-1 に配置して災害発生時にフェイルオーバーできる状態を求めている。
さらに金融規制当局への報告期限が迫っており、運用チームの工数を増やさず迅速に切り替えを行える仕組みが必要となった。
ビジネス上は顧客離脱リスクの低減、技術面では需要連動のスケールアウトと RPO 数分以内のデータ同期を同時に満たすことが条件である。
ソリューションアーキテクトはこれらの制約を踏まえ、インフラレベルでどの AWS 構成を採用すべきか？

# Options

A. us-east-1 と us-west-1 に同一構成の VPC・ALB・EC2 Auto Scaling グループを用意し、Route 53 のフェイルオーバールーティングとヘルスチェックでプライマリ障害時のみ待機系 ALB へ切り替える。
B. 2 つのリージョン の VPC 間で VPC ピアリングを設定し、us-east-1 側に 1 つの Application Load Balancer (ALB) を作成して両方の VPC の複数 AZ をカバーする。両リージョンの EC2 Auto Scaling グループを同一 ALB のターゲットとして登録し、単一エンドポイントから全トラフィックを処理する。
C. us-east-1 と us-west-1 の両方に同一構成の ALB と EC2 Auto Scaling グループをデプロイし、どちらも常時アクティブなリージョンとして扱う。Route 53 では 2 つの ALB をレコードとして登録し、レイテンシーベースルーティングポリシーを設定して、利用者に最も近いリージョンへ自動的にトラフィックを送る。障害時にはヘルスチェックによって一方のリージョンに集約されることを想定する。
D. us-east-1 と us-west-1 に個別の VPC を作成し、それぞれに複数 AZ 構成の Application Load Balancer (ALB) と EC2 Auto Scaling グループをデプロイする。Amazon Route 53 のホストゾーンに 2 つの ALB 用レコードを登録し、各レコードにヘルスチェックのみを関連付ける。ルーティングポリシーはデフォルト設定のままとし、ヘルスチェック結果に応じてクエリごとにいずれかの ALB へトラフィックを振り分ける。
